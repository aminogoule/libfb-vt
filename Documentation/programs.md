# Программы

`[vt]` — это номер VT с отсчётом от 1 (7 == ttyv6); опустите или укажите 0, чтобы
автоматически выбрать первый свободный VT.

- `ppm2fb image.ppm` — одноразово: нарисовать изображение один раз на текущей консоли.
- `fbshow  image.ppm [vt]` — постоянный просмотрщик (опрос; захватывает свободный VT).
- `server  [-e команда] [-w обои.ppm] [-k] [vt]` — **компоновщик (compositor)**: эксклюзивно
  владеет VT, слушает `AF_UNIX`-сокет, выдаёт каждому клиенту разделяемый буфер пикселей на
  «поверхность» (окно), композитит все окна (фон рабочего стола + рамка + заголовок + пиксели
  клиента) и раздаёт ввод с клавиатуры сфокусированному (верхнему) окну. `-e команда` — после
  захвата VT и поднятия сокета запустить `команду` (через `/bin/sh -c`) как первого клиента.
  `-k` — читать клавиатуру как сырые `K_CODE`-скан-коды вместо `K_XLATE` (см.
  [input.md](input.md#клавиатура)); по умолчанию выключено.
- `term    [rows] [cols]` — **графический терминал** как клиент компоновщика: своего VT/framebuffer
  не имеет, общается по протоколу, получает разделяемый буфер под окно, форкает shell в pty
  (`forkpty`) и эмулирует VT100-подобный терминал в этот буфер встроенным шрифтом 8×16.
  Размер по умолчанию 25×80, обрезается под выданный размер. Собирается один бинарник,
  не зависящий от бэкенда (линкуется с `-lutil`). Полный список поддерживаемых
  escape-последовательностей — ниже, в разделе [«Что умеет term»](#что-умеет-term).
- `cube    [vt]` — демо: вращающийся сплошь затенённый 3D-куб в окне 320x200
  (тот же скелет сервера; программный рендерер, работает на любом бэкенде). Требует `-lm`.
- `glcube  [vt]` — тот же куб, но нарисован через `mgl` (см. [ниже](#mgl)) — реальными
  `mgl_begin/mgl_vertex3f/mgl_rotate/...` вместо ручной тригонометрии. Требует `-lm`.
- `cubewin [width] [height]` — тот же mgl-куб, но как **клиент компоновщика** (как `term`),
  а не отдельный владелец VT: HELLO/CREATE_SURFACE по протоколу, рендер каждого кадра прямо
  в выданный разделяемый буфер окна, `FBVT_COMMIT`. Не открывает framebuffer и не претендует
  на VT, поэтому не гоняется с компоновщиком за консоль — в отличие от `cube`/`glcube`, которые
  каждый сам захватывает VT (`vtcon_acquire`) и годятся только как самостоятельные демки, не
  одновременно с `server`. Размер окна по умолчанию 320x200. `q`/`Q`/Esc или закрытие окна
  компоновщиком — выход. Собирается один бинарник, не зависящий от бэкенда. Требует `-lm`.

## Сборка

    make
    # или, например, только клиент терминала:
    make term
    # server напрямую (выберите бэкенд — .svga / .vga / .fb):
    make server.svga

## Запуск: компоновщик + терминал

Требуется root, на консоли vt(4) (`sysctl kern.vty` должен сообщать `vt`).

Компоновщик монопольно захватывает VT, поэтому запустить клиент «руками» с той же
консоли уже не выйдет — на заблокированном VT нет доступного shell. Два рабочих способа:

    # 1) пусть сервер сам поднимет первый терминал (проще всего):
    sudo ./server.svga -e ./term

    # 2) или запустить клиента отдельным процессом в той же командной строке:
    sudo sh -c './server.svga & sleep 1; ./term; wait'

    # обои на весь экран (PPM, растягивается nearest-neighbour):
    sudo ./server.svga -w ./wall.ppm -e ./term

    # окно с mgl-кубом вместо/вместе с терминалом (тот же принцип клиента):
    sudo ./server.svga -e ./cubewin
    # либо из RMB-меню рабочего стола -> "New terminal" запустить server с -e,
    # либо просто отдельным процессом, как во варианте (2) выше, заменив ./term на ./cubewin

Клавиатура уходит в верхнее (сфокусированное) окно; всё, что печатаете, попадает в shell.
Компоновщик завершается по `SIGINT`/`SIGTERM`/`SIGHUP` **или** автоматически, когда отключился
последний подключавшийся клиент (например, вы набрали `exit` в терминале). `server` отказывается
переключать VT (`Alt+Fn`, `Ctrl+Alt+Fn`, `Alt+стрелки`) во время работы; см.
[backends.md](backends.md#переключение-vt-во-время-работы-сервера).

Мышь и клавиатура (клики, скролл, xterm mouse-reporting, K_CODE) — [input.md](input.md).
Смена разрешения в реальном времени — [backends.md](backends.md#смена-разрешения-в-реальном-времени).
Не грузится/чёрный экран — [troubleshooting.md](troubleshooting.md).

## Что умеет term

`$TERM=xterm-256color`. Реализовано:

- печать с переносом/прокруткой;
- C0-управление (CR/LF/BS/TAB/BEL);
- CSI-перемещения курсора и очистка (`A B C D H f G d J K`);
- вставка/удаление символов в строке — `@` (ICH) и `P` (DCH) — нужны readline/libedit при
  правке строки истории по месту (Up за командой из истории, стрелки на середину строки, Del
  или ввод символа посередине);
- IRM — Insert/Replace Mode, `CSI 4h`/`CSI 4l` (**не** DEC-приватный, без `?`): readline/libedit
  включают его перед каждым вставляемым символом вместо ICH — `put_char()` в этом режиме
  сдвигает остаток строки вправо вместо перезаписи;
- область прокрутки DECSTBM (`r`) + `IL`/`DL`/`SU`/`SD`;
- альтернативный экран (DECSET `47`/`1047`/`1049` — vim/mc не оставляют мусор на экране после
  выхода);
- курсорные клавиши в режиме приложения (DECCKM) и функциональные клавиши F1–F12,
  переведённые под то, что реально знает `xterm-256color`-terminfo;
- цвета SGR — 16-цветные (`0/1/7/30–37/40–47/90–97/100–107/39/49`), 256-цветные и truecolor
  (`38/48;5;n` и `38/48;2;r;g;b`, хранятся как честный 24-битный RGB на ячейку);
- показ/скрытие курсора (DEC `?25`), заголовки окна (OSC `0`/`2`);
- отчёты мыши (DECSET `1000`/`1002`/`1006`) и wheel-to-arrow-keys для alt-screen приложений
  без своего mouse-reporting — подробности в [input.md](input.md#мышь-в-term).

Достаточно для интерактивного shell и разумно точного показа изображений через ANSI-арт
вьюеры (`chafa`, `catimg`, ...); не полный клон xterm (нет модификаторов клавиш, bracketed
paste и т.п.).

## mgl

`mgl.c` / `mgl.h` — маленькая **программная** реализация подмножества OpenGL 1.x в
immediate-mode стиле (`glBegin/glVertex3f/glColor3f/glEnd`, стек матриц
`MODELVIEW`/`PROJECTION`, `mgl_translate/rotate/scale/perspective/ortho`), растеризующая
в приватный внутренний back-буфер. Никакого GPU/DRM/KMS/vmwgfx: чистый CPU-рендер с
Z-буфером и Gouraud-закраской по треугольнику (без perspective-correct интерполяции, без
текстур, без `GL_LIGHTING` — цвет каждой вершины задаётся приложением через `mgl_color3f`,
как во `glcube.c`/`cubewin.c`). По умолчанию **двусторонний** (backface culling выключен) —
включается через `mgl_enable_cull_face(1)`, если приложение само гарантирует единый обход
вершин по всем граням (общая таблица граней куба `CUBE_F`, унаследованная от `cube.c`, этого
не гарантирует, поэтому демо-кубы рисуются двусторонними и полагаются только на Z-буфер).

**Двойная буферизация**: всё рисование (`glBegin`/`glEnd`, `mgl_clear`) идёт во внутренний
back-буфер `mgl`, а не в целевой буфер напрямую; `mgl_swap_buffers()` одним проходом
копирует его в целевой буфер (`fb_drawbuf()` для `glcube`/`cube`, разделяемый буфер окна
для `cubewin`) непосредственно перед `fb_flip()`/`FBVT_COMMIT` — так читающая сторона
(compositor или сам `fb_flip`) никогда не видит буфер наполовину отрисованным. И `mgl_clear`,
и `mgl_swap_buffers` работают только в пределах текущего `mgl_viewport()`, а не всего
целевого буфера — это важно для `glcube.c`, где mgl-мишень — это буфер *всего экрана*
(рабочий стол и рамка окна дорисованы вручную за пределами вьюпорта куба).

Это **первый шаг** — «программный GL», достаточный, чтобы существующий и новый софт мог
рисовать через привычный API матриц/вершин. Настоящее аппаратное ускорение через
SVGA3D-протокол VMware (guest-backed surfaces, command buffer, шейдеры) — отдельная,
существенно большая задача, запланированная на будущее (отдельная ветка/сессия), не
входит в этот шаг.

`mgl` не привязан к тому, *куда* он рисует: `glcube` линкует его вместе с бэкендом и
рисует прямо в `fb_drawbuf()` (владея VT самостоятельно, как `cube`), а `cubewin` линкует
его вообще без бэкенда и рисует в разделяемый буфер окна, выданный компоновщиком (как
`term`) — с точки зрения `mgl` это один и тот же `uint8_t*`/`pitch`/`width`/`height`.
Отдельного `libmgl` пока нет — просто исходник, линкуемый в нужный бинарник.
